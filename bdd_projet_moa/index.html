🚀 Priorisation & Sprints

Répondre au besoin de votre chef : "Bouger des blocs pour re-prioriser"

La solution est de créer une page dédiée "Backlog & Sprints" avec une vue Kanban. Cette vue permettra de visualiser et de modifier la priorité des demandes par un simple glisser-déposer.

Mise en Place de la Vue Kanban de Priorisation

  1. Ajouter une propriété `Sprint` : Dans votre base de données "Pipeline Global", ajoutez une propriété de type `Sélection` nommée "Sprint".
    • Options : 🧊 Backlog, ▶️ Sprint Actuel, ⏭️ Sprint Suivant.
  2. Créer une nouvelle page "Backlog & Sprints" : C'est ici que vous gérerez votre roadmap.
  3. Créer la vue Kanban : Sur cette page, créez une vue liée de votre base de données. Choisissez la disposition "Tableau" (Kanban) et groupez par la propriété `Sprint`.
  4. Prioriser par glisser-déposer : Vous pouvez maintenant "bouger les blocs" !
    • Déplacer une carte d'une colonne à l'autre (ex: de "Backlog" à "Sprint Actuel") pour planifier le travail.
    • Réorganiser les cartes verticalement dans la colonne "Sprint Actuel" pour définir l'ordre de priorité de développement.

🔑 Processus Clés de Qualification

🔴 Triage vs. Analyse 🔵

Le Triage est un filtre rapide (1-5 min) pour valider la clarté et l'unicité. L'Analyse est une investigation fonctionnelle approfondie.

🔌 GAP vs. Story ✨

Un GAP est un rattrapage par rapport à un ancien outil. Une Story est une pure nouvelle fonctionnalité.

🎯 Pilotage Stratégique

Comment Construire vos Indicateurs de Priorisation

⭐ Mettre en place le Score RICE

Oui, il s'agit bien d'une propriété de type 'Formule' dans Notion. Voici comment la configurer :

  1. Assurez-vous d'avoir les 3 propriétés numériques : `Impact` (nombre de 1 à 5), `Confiance` (nombre de 0 à 100), `Effort` (nombre en jours-homme).
  2. Créez une nouvelle propriété `Score RICE` de type `Formule`.
  3. Cliquez sur "Modifier la formule" et collez la formule suivante :
(prop("Impact") * (prop("Confiance") / 100)) / prop("Effort")

Note : La formule divise la Confiance par 100 pour la transformer en multiplicateur (ex: 80% devient 0.8). Si l'Effort est 0, la formule renverra une erreur ; assurez-vous de toujours estimer l'effort (minimum 0.5 pour une petite tâche).

⏱️ Mettre en place l'Analyse des Délais

Cet indicateur, aussi appelé "Lead Time", mesure le temps total entre la création d'une demande et sa résolution. Il se calcule aussi avec une formule.

  1. Vous avez déjà les propriétés `Date de création` et `Date de résolution`.
  2. Créez une nouvelle propriété `Délai de Résolution (jours)` de type `Formule`.
  3. Cliquez sur "Modifier la formule" et collez la formule suivante :
if(prop("Statut") == "📦 Terminé", dateBetween(prop("Date de résolution"), prop("Date de création"), "days"), null)

Note : Cette formule n'affichera un résultat que si le statut est "Terminé", ce qui garde votre base de données propre. Dans une vue de type "Tableau", vous pourrez ensuite cliquer en bas de cette colonne pour calculer automatiquement la moyenne des délais de résolution.

🛠️ Plan de Déploiement

Phase 1 : Construction du Socle

Créer la base de données Pipeline Global et configurer toutes les propriétés de la section Taxonomie.

Phase 2 : Création de l'Espace de Travail

Construire la page d'accueil Hub Projet, les templates et les boutons. Créer la page et la vue Kanban de Priorisation & Sprints.

Phase 3 : Migration & Formation

Migrer un petit lot de tickets existants en utilisant le guide de migration. Former le groupe pilote (votre assistant) sur le nouveau processus.

Conseil d'Expert : Le retour du groupe pilote est crucial. Ajustez la taxonomie ou les templates si nécessaire AVANT de migrer la totalité des données.

Phase 4 : Déploiement Complet

Après validation du pilote, migrer l'ensemble des tickets restants. Communiquer officiellement le lancement du nouveau Hub à toutes les parties prenantes et archiver les anciens espaces.
